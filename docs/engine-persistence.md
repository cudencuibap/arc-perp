# Engine State Persistence (Phase 3)

The matching engine in-memory state — open positions, balances (incl. realized PnL), resting orderbook, trade tail — survives restart by snapshotting to a single JSON file with atomic writes. Without this, every engine restart resets realized PnL and orderbook to empty, which is unworkable for autonomous agents that may run for days.

## What persists

| Field | Reason |
|---|---|
| `positions` | Money at risk; must survive crash |
| `balances` | Realized PnL accumulators (bot sim + persisted human history) |
| `books` (resting orders) | User limit orders; **`seed-mm-*` entries dropped on reload** (regenerated by `seedLiquidity` interval every 700ms) |
| `trades` | Tail-capped at 1000 entries AND ts ≥ now − 24h (whichever is smaller) — needed for `volume24h` calc |
| `markets` | Snapshotted but considered stale on load; warmup gate refuses orders until fresh marks arrive |
| `agentIds` | Small set used by World heatmap |

Not persisted: WS connections, event emitter listeners, `recentEvents` ring buffer (regenerated as new events flow).

## State file

Path: `services/matching-engine/data/engine-state.json` (git-ignored via root `data/`).

Shape:

```json
{
  "version": 1,
  "savedAt": "2026-06-06T13:25:00.000Z",
  "books": { "BTC-PERP": { "bids": [/* Order */], "asks": [/* Order */] }, ... },
  "balances": [/* Balance */],
  "positions": [/* Position */],
  "trades": [/* trimmed Trade tail */],
  "markets": { "BTC-PERP": {/* MarketMeta */}, ... },
  "agentIds": [/* string */]
}
```

Atomic write: `writeFile(<path>.tmp)` then `rename(<path>.tmp, <path>)`. A crash mid-write either leaves the previous snapshot intact (tmp orphan, real file untouched) or the new snapshot fully visible. `fs.rename` is atomic on the same filesystem on both POSIX and Windows.

## Snapshot trigger

Three signals collapse into a single serialized persist chain so concurrent writes cannot race the rename:

1. **Debounced fill events** — after `trade`, `balance`, `position`, or `liquidation` events from the engine, a 500ms timer is (re-)armed. A burst of fills coalesces into one disk write at the end of the burst. `mark` events skip — they fire 15× per second across the three symbols and the loss of market state on crash is recovered in milliseconds by the market-data publisher.
2. **30s heartbeat** — `setInterval` triggers a persist regardless of activity. Catches drift in `markets` and any missed debounce.
3. **SIGTERM / SIGINT** — handler cancels any pending debounce and awaits a final flush before `process.exit(0)`. Graceful shutdown loses zero state.

Worst-case write rate during sustained heavy trading: ~2 Hz (debounce ceiling). Worst-case state loss on `SIGKILL`: up to 500ms of fills, or 30s during idle.

Trigger intervals are tunable:

```
ENGINE_SNAPSHOT_DEBOUNCE_MS=500
ENGINE_SNAPSHOT_HEARTBEAT_MS=30000
```

Set `ENGINE_PERSIST_ENABLED=false` to disable persistence entirely (useful for stateless e2e fixtures or to A/B reproduce a bug).

## Boot sequence

1. `loadState(statePath)` reads + version-checks the file. Returns `null` if missing (fresh boot). Throws on corrupt or unknown-version content; wrapper logs and exits(1) so operators choose: investigate or delete to reset.
2. If a snapshot was loaded, `engine.restoreInternalState(toInternalState(parsed))` hydrates Maps and Set, drops `seed-mm-*` orders from books, and re-sorts each book to enforce the price-time priority invariant.
3. `restoreTs = Date.now()` — used by the warmup gate.
4. `serverListen()` binds port 4101.

## Warmup gate

After restore, the engine has stale `position.markPrice` values from the pre-restart session. A position that should have been liquidated during downtime might still be open. Until every required symbol receives at least one fresh mark from market-data (post `restoreTs`), the `POST /orders` handler short-circuits with:

```http
HTTP/1.1 503 Service Unavailable
Retry-After: 1
Content-Type: application/json

{
  "error": "engine_warming",
  "code": "ENGINE_WARMING",
  "retryAfter": 1,
  "message": "Engine warming up after restart — waiting for fresh marks from market-data."
}
```

Autonomous agents branch on `code === "ENGINE_WARMING"` and retry after the indicated delay. Typical warmup window is ~600ms (3 symbols × 200ms `publishMs` from market-data).

Once `markets.every(m => m.ts > restoreTs)` becomes true, `restoreTs` is cleared and the gate becomes a no-op for the lifetime of the process.

## Mark-gap safety — atomic liquidation on first updateMark

Phase 3 adds one safety property to the core engine that closes the mark-gap exploit:

> When `updateMark` arrives, after `revalue` updates each affected position's `liquidationPrice`, any position whose new mark has crossed that liquidationPrice is immediately liquidated via `this.liquidate(traderId, symbol)` — before the `mark` event is emitted, before any new order can be placed.

This means: a user with a 10× long at 100,000 (liquidationPrice ≈ 95,500) cannot survive a price drop to 90,000 by being lucky enough for the engine to be offline during the move. As soon as the engine reboots and receives the first fresh mark, the position is force-closed with the standard liquidation penalty.

Idempotent against `services/liquidation-engine`: that auxiliary service polls `/state` every 1.2s and POSTs to `/liquidations`; `engine.liquidate(traderId, symbol)` returns `false` when the position has already been removed, so a stray poll is a no-op.

## Known limitations

1. **No reorg handling** — Arc testnet reorgs are rare but possible. The persisted state assumes a linear chain; on a reorg the snapshot may diverge from on-chain truth. Reset by deleting `engine-state.json`.
2. **No cross-instance persistence** — single-node design. A multi-instance deployment would need shared backing storage (redis, sqlite). Out of scope for testnet.
3. **`recentEvents` ring buffer not restored** — World heatmap is empty on boot until ~1s of new events accumulate. Cosmetic.
4. **liquidation-engine becomes redundant** — the engine-side atomic liquidation in `updateMark` makes the auxiliary poller a no-op. Safe to keep running as belt-and-suspenders; can be removed in a cleanup phase.
5. **JSON state file size** — snapshots grow with order count + position count + trade tail. At 1000 trades + dozens of positions + ~50 orders, expect ~50–100 KB per write. Sustained 2 Hz writes ≈ 200 KB/s disk. Acceptable for testnet; consider binary format if scaling.

## Verification

`packages/core/src/__tests__/persistence.test.ts` covers:

- `getInternalState` returns deep clones (mutation of returned state does not affect engine)
- `restoreInternalState` round-trips state through `engine.state()` identically
- `updateMark` after restore recomputes unrealized PnL against the new mark
- **First `updateMark` after restore auto-liquidates an underwater position** — the critical mark-gap safety property
- Still-solvent position is left alone

`services/matching-engine/src/__tests__/persist.test.ts` covers:

- `serialize` trims trades to tail 1000 and drops anything older than 24h
- `deserialize` drops `seed-mm-*` orders from books
- Schema version mismatch and malformed JSON both throw
- `saveState` + `loadState` round-trip is identical post-JSON normalization
- Atomic write leaves no `.tmp` orphan on success
- `loadState` returns `null` for a missing file
- `toInternalState` hydrates Map and Set types correctly without redundant JSON round-trip
